# Demo Mode

BudPomich Demo Mode дає незареєстрованому відвідувачу окремі заповнені кабінети клієнта, майстра та підрядника. Усі три ролі використовують одну поточну Supabase Auth-сесію й один анонімний `user_id`.

## Дані

`demo_templates` містить версійовані початкові стани кожної ролі. Шаблони читаються під час першого створення рольової демосесії, reset та відновлення простроченої сесії.

`demo_sessions` містить приватний стан конкретного відвідувача для конкретної ролі. Для одного користувача може існувати не більше однієї строки на роль. Зміни клієнта, майстра й підрядника зберігаються незалежно.

Активна демосесія діє 24 години. Якщо вона прострочена, застосунок відновлює ту саму строку з актуального активного шаблону, оновлює `template_version`, `last_accessed_at` та `updated_at` і продовжує `expires_at` на 24 години. Новий Auth-користувач і нова строка `demo_sessions` при цьому не створюються.

## Автоматичне очищення

Supabase Cron щодня виконує дві серверні задачі:

- `budpomich-clean-expired-demo-sessions` — о 03:10 UTC видаляє `demo_sessions`, прострочені понад 7 днів;
- `budpomich-clean-old-anonymous-users` — о 03:30 UTC видаляє тільки записи `auth.users` з `is_anonymous is true`, створені понад 30 днів тому.

Видалення анонімного користувача каскадно видаляє пов'язані демосесії. Зареєстровані користувачі (`is_anonymous = false`) не видаляються.

## Ізоляція та безпека

Demo Mode працює через поточну Supabase-сесію та RLS. `user_id` не приймається з URL або браузерного payload. Браузер надсилає лише дозволені типізовані дії, а не повний state. Service role у клієнті не використовується.

Демонстраційні дані зберігаються лише у `demo_sessions.state` і не змішуються з реальними `bookings`, `messages`, профілями, командами чи проєктами. Reset змінює тільки демосесію вибраної ролі.
